# Agent File Systems

Dust agents interact with files through four distinct file systems, each with different storage
backends, access patterns, and scoping rules. This document describes each one.

## Overview

| File System | Scope | Storage | Agent Access | Sandbox Access |
|---|---|---|---|---|
| **Conversation Files** | Single conversation | GCS | MCP tools (`conversation_files`) | gcsfuse mount at `/files/conversation` |
| **Project Files** | All conversations in a project | GCS + Core data source | MCP tools (`project_manager` + `conversation_files`) | Not directly mounted |
| **Data Source Files** | Workspace-wide (per data source) | Core API | MCP tools (`data_sources_file_system`) | Not mounted |
| **Sandbox Files** | Single conversation (ephemeral) | E2B container local disk | Via sandbox execution | Direct filesystem access |

## 1. Conversation Files

**Scope:** A single conversation.
**Use cases:** `conversation`, `tool_output`

Files uploaded by users or generated by tools within a conversation. These are the only files
directly accessible in the sandbox filesystem.

### Storage

Files are stored in **Google Cloud Storage (GCS)** under a structured path:

```
w/<workspaceId>/conversations/<conversationId>/files/<fileName>
```

The GCS bucket is the private uploads bucket (`fileStorageConfig.getGcsPrivateUploadsBucket()`).

### Agent access (MCP)

The `conversation_files` server provides three tools:

- **`list`** - List all files attached to the conversation.
- **`cat`** - Read file contents with offset/limit pagination and optional grep filtering.
- **`semantic_search`** - Semantic search within conversation file contents.

When a conversation belongs to a project, these tools also surface project context files (see below).

### Sandbox access (gcsfuse)

Conversation files are mounted into the sandbox at **`/files/conversation`** via gcsfuse. The
mount sequence:

1. **Token minting** - A downscoped GCS token is created via Google STS, restricted to the
   conversation's prefix only (using Credential Access Boundaries).
2. **Token server** - A netcat-based HTTP server at `localhost:9876` serves the token to gcsfuse.
3. **gcsfuse mount** - Mounts the GCS bucket filtered to the conversation prefix with `--only-dir`.

Key properties:
- Read/write: files written to `/files/conversation` persist to GCS.
- Token refresh: for already-running sandboxes, only `/tmp/token.json` is refreshed (no remount).
- A `.mount-pending` sentinel file signals the mount is not yet ready.
- `.processed.*` files are internal text-extraction artifacts, filtered from user-facing listings.

**Key files:**
- `front/lib/api/sandbox/gcs/mount.ts` - Mount logic
- `front/lib/api/sandbox/gcs/token.ts` - Downscoped token minting
- `front/lib/api/files/mount_path.ts` - GCS path helpers
- `front/lib/api/actions/servers/conversation_files/` - MCP server

## 2. Project Files

**Scope:** Shared across all conversations within a project.
**Use case:** `project_context`

Projects are spaces with `kind: "project"`. They provide a shared file context that persists
across all conversations in the project.

### Storage

Project files use the same GCS-backed `FileResource` storage as conversation files, but with:
- `useCase: "project_context"`
- `useCaseMetadata.spaceId` pointing to the project space

Additionally, project files are **indexed into a Core data source** for semantic search:
- Documents are upserted under a "Context" folder in the project's data source.
- Tables (CSV/TSV) are also indexed for structured querying.

### Agent access (MCP)

The `project_manager` server provides:

- **`add_file`** - Create a new file (from text content or by copying an existing conversation file).
- **`update_file`** - Replace content of an existing project file.
- **`edit_description`** - Modify the project description.
- **`get_information`** - Fetch project URL, description, and file list.
- **`list_unread`** - Search for unread conversations in the project.

Project files are also surfaced through the `conversation_files` server's `list` and
`semantic_search` tools when the conversation belongs to a project.

### Sandbox access

Project files are **not directly mounted** in the sandbox. They are accessible to the agent
through MCP tools only. The agent can read project file contents via `conversation_files.cat`
or `project_manager.get_information` and write them to the sandbox if needed.

**Key files:**
- `front/lib/api/actions/servers/project_manager/` - MCP server
- `front/lib/api/projects/context.ts` - Project file indexing into Core data source
- `front/lib/resources/file_resource.ts` - `listByProject()` method

## 3. Data Source Files

**Scope:** Workspace-wide, scoped by data source views and permissions.

Files from connected data sources (Google Drive, Notion, Confluence, Slack, GitHub, etc.) and
manually uploaded folders. These are stored and indexed by the Core API as content nodes in a
hierarchical tree structure.

### Storage

Content is stored in the **Core API** as nodes with three types:
- **Documents** - Files with extractable text content.
- **Folders** - Containers in the hierarchy.
- **Tables** - Structured/tabular data.

Each node has `node_id`, `parent_id`, and a `parents` ancestry chain, creating a filesystem-like
tree.

### Agent access (MCP)

The `data_sources_file_system` server provides Unix-like tools:

- **`list`** - List direct children of a node (`ls`). Pass no `nodeId` to list available data sources.
- **`find`** - Find nodes by title from a starting node (`find`).
- **`cat`** - Read document text content by `nodeId`, with offset/limit/grep support (`cat`).
- **`semantic_search`** - Full-text semantic search scoped to specific nodes.
- **`locate_in_tree`** - Show complete path from a node to the data source root.
- **`find_tags`** - Find available tags within data sources (when tags are dynamic).

Access is scoped through `DataSourceView` which can filter by:
- `parentsIn` - Restrict to specific folder subtrees.
- `tags` - Tag-based filtering.
- Workspace permissions and group membership.

### Sandbox access

Data source files are **not mounted** in the sandbox. They are accessible to the agent exclusively
through MCP tools. The agent can read content via `cat` and write it to the sandbox filesystem if
the user needs it processed there.

**Key files:**
- `front/lib/api/actions/servers/data_sources_file_system/` - MCP server
- `front/types/core/content_node.ts` - Node type definitions
- `front/types/core/data_source.ts` - Document/folder types

## 4. Sandbox Files

**Scope:** A single conversation (ephemeral container storage).

The sandbox is an isolated Linux container (E2B) where agents execute code. It has its own
local filesystem that is independent of the other file systems.

### Storage

- **Container-local storage**: `/tmp`, `/home/user`, and other standard Linux paths. This storage
  is ephemeral and lost when the sandbox is paused (after ~10min idle) or destroyed (after ~24h).
- **Mounted conversation files**: `/files/conversation` (see section 1). This is the only
  persistent storage accessible from the sandbox.

### Agent access

The `sandbox` server provides:

- **`bash`** - Execute shell commands in the sandbox. Can read/write any path.
- **`describe_toolset`** - List available CLI tools and libraries.

### Sandbox lifecycle

- **Running** - Active, all files accessible.
- **Sleeping** - Paused after ~10min idle. Container state preserved but not executing.
  Woken on next use, GCS token refreshed.
- **Deleted** - Destroyed after ~24h idle. All container-local files lost. Conversation files
  in GCS persist.

The sandbox comes with pre-installed tools: Python 3 with data science libraries (pandas, numpy,
matplotlib, plotly, etc.), Node.js/TypeScript, and system utilities (git, curl, jq, pandoc,
imagemagick, ffmpeg, etc.).

### Skill attachments

Files attached to skill configurations (`useCase: "skill_attachment"`) are synced to the sandbox
at `/dust/skills/<skill-name>/<filename>`, making them available to code execution within the
skill context.

**Key files:**
- `front/lib/api/sandbox/providers/e2b.ts` - E2B provider
- `front/lib/api/sandbox/image/registry.ts` - Base image configuration
- `front/lib/resources/sandbox_resource.ts` - Sandbox lifecycle management
- `front/temporal/sandbox_reaper/` - Automatic sleep/destroy workflows
- `front/lib/api/actions/servers/sandbox/` - MCP server

## How files flow between systems

```
User uploads file ──> GCS (conversation files)
                        │
                        ├──> gcsfuse ──> Sandbox /files/conversation (direct access)
                        │
                        └──> conversation_files MCP tools (list, cat, search)

Project file added ──> GCS (project_context) + Core data source
                        │
                        ├──> project_manager MCP tools (add, update, get_info)
                        │
                        └──> conversation_files MCP tools (when in project context)

Data source synced ──> Core API (content nodes)
                        │
                        └──> data_sources_file_system MCP tools (list, find, cat, search)

Agent runs code ──> Sandbox local filesystem (ephemeral)
                        │
                        ├──> Can read /files/conversation (gcsfuse)
                        │
                        └──> Can write to /files/conversation (persists to GCS)
```

The key architectural insight: **only conversation files bridge the MCP/sandbox boundary**.
Data source files and project files are accessed through MCP tools by the agent's reasoning
layer but are not available as filesystem paths in the sandbox. If the agent needs to process
them with code, it must first read the content via MCP tools and write it to the sandbox.
